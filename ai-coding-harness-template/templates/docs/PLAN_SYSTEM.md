# PLAN_SYSTEM.md

> 단순하지 않은 작업은 코드를 만지기 전에 plan을 만든다. plan은 사용자 승인 후 실행한다.

## 언제 plan을 작성하는가

- 2 파일 이상을 동시에 변경하는 경우
- 새 도메인/모듈/엔티티를 추가하는 경우
- 마이그레이션 또는 데이터 이행이 포함되는 경우
- 의존성 추가/업그레이드가 포함되는 경우
- UI 흐름이 추가/변경되는 경우

## plan 형식

```markdown
# Plan: <작업 제목>

## 목표
<한 문단으로 무엇을 왜 하는지>

## 영향 범위
- 변경 파일: ...
- 영향 도메인: ...
- 비영향 (의도적으로 손대지 않음): ...

## 단계
1. ...
2. ...

## 검증 방법
- 작업 중 반복 확인: <빠른 검증 명령>
- 완료 확인(최종): <완료 검증 명령>
- 수동 확인: ...

## 롤백
<문제 시 어떻게 되돌릴지>
```

## plan 문서 (작업 노트)

plan을 작성할 때 그 내용을 담는 **md 파일을 생성**하고, 작업 진행 중 메모·단계 체크·결정 사항을 그 파일에서 관리한다.

- **위치**: `docs/plans/<이슈번호 또는 주제>.md`
- **git 미추적**: 개인 작업용 문서다. `docs/plans/` 는 `.gitignore` 대상. **git 에 올리지 않는다.**
- **작업 중**: 단계 진행·메모·결정을 이 파일에서 갱신 (영구 규칙은 적지 않는다 — 그건 해당 카테고리 문서로).
- **완료 시**: 작업이 끝나면 plan 내용을 **제거**한다 (파일 비우기 또는 삭제).

## 절차

1. plan 초안 작성 + `docs/plans/` 에 plan md 생성 → 사용자 검토.
2. 승인 후 단계별로 구현 + 단계마다 빠른 검증 (진행 상황은 plan md 에 기록).
3. 완료 시 완료 검증 통과 확인 후 plan md 내용 제거. 이슈 상태 갱신은 사용자가 직접.

## 작업 유형별 체크포인트

### 버그 수정
- 재현 단계 (1, 2, 3...) / 기대값 vs 실제값 / 영향 파일(추정) / 회귀 우려 인접 기능

### 신규 화면 추가
- 참조 화면 (유사 패턴 1~2개) / URL·라우팅 / 각 레이어 파일 위치 / 데이터 모델(테이블·컬럼) / UI 컴포넌트 선택

### 리팩터링
- 변경 전 → 변경 후 패턴 명시 / 비기능 변경 없음을 확인하는 방법 / 영향 범위

### 의존성 업그레이드
- 릴리스 노트 요약(호환성 깨짐·취약점·동작 변경) / 영향 파일 / 회귀 우려 / **사용자 승인 필수**

### DB / 쿼리 변경
- 변경 요지 / 페이징·인덱스 영향 / 바인딩 규칙(파라미터 바인딩 사용, 문자열 연결 금지 → `SECURITY.md`)

### 외부 API / 실시간 통신
- 요청·응답 스펙 / 인증(세션·토큰) / 재시도·타임아웃 / API 문서 갱신 여부 / 권한 체크 누락 주의

## 금지

- plan 없이 다중 파일 리팩터링 시작 금지.
- plan에 없는 부수 작업 끼워넣기 금지(별도 plan으로 분리).
