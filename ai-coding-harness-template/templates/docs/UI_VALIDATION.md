## 사전 확인 — 브라우저 자동화 MCP 로드 여부

작업 시작 시점에 브라우저 자동화 MCP 가 세션에 로드돼 있는지부터 확인한다.

- **로드된 경우**: 아래 "검증 단계" 3~5번을 AI 가 자동 수행.
- **로드되지 않은 경우**: AI 는 빌드 통과 / 서버 기동 / 코드 정합성까지만 점검하고,
  브라우저 확인·콘솔·네트워크·스크린샷은 **"수동 확인 필요"** 로 사용자에게 명시 위임.
  단정적으로 "UI 정상 동작 완료" 라고 보고 금지.

## 검증 단계

1. **빌드 통과** — 완료 검증 명령 그린 (작업 중 반복 확인은 빠른 검증 명령).
2. **서버 기동** — 실행 후 시작 로그(에러 없음) 확인. 백그라운드 실행 권장.
3. **브라우저 확인** — 브라우저 자동화 MCP 로:
   - 기본 흐름(golden path): 페이지 진입 → 주요 액션 → 결과.
   - 엣지 케이스: 빈 데이터, 권한 없는 사용자, 긴 텍스트 등.
   - 회귀 확인: 인접 화면 1~2 개 함께 점검.
4. **콘솔 확인** — 브라우저 Console 에 에러/경고 없는지.
5. **네트워크 확인** — 4xx / 5xx 응답이 있는지.
6. **스크린샷** — 결과 화면 스크린샷 1~2 장 첨부해서 보고.

## AI 가 자동으로 처리하는 범위

- 페이지 네비게이션, 폼 입력, 버튼 클릭 등 기본 시나리오.
- Console / Network 로그 수집. 스크린샷 캡처.
- 인코딩 깨짐 등 시각적으로 명백한 문제 감지.

## AI 가 사용자에게 넘기는 부분 (단정 금지)

- **로그인 자격 증명이 필요한 경로** — 사용자 권한별 시나리오는 사용자 확인.
- **데이터 정합성 / 도메인 의미** — 비즈니스 판단이 필요한 정확성.
- **레이아웃·디자인 미세 조정** — 픽셀 단위 디자인 의도.
- **장시간 시나리오** — 배치/예약/장시간 폴링 등.

## 보고 양식

```
[UI 검증 결과]
- 빌드: PASS / FAIL
- 기동: PASS / FAIL
- 자동 확인 시나리오:
  · GET /xxx/list → 200, 행 N개 노출, 콘솔 에러 없음
  · 페이징 클릭 → 200, 데이터 갱신 확인
- 스크린샷: <첨부>
- 사용자 확인 필요 항목: ...
- 미확인 (이유): ...
```

## 금지

- 빌드만 통과시키고 "UI 정상 동작 완료" 라고 보고 금지.
- 자동화로 한 번 통과했다고 모든 시나리오를 검증한 것처럼 보고 금지.
- 인코딩 깨짐 의심되면 반드시 브라우저에서 직접 확인.
- 브라우저 자동화 MCP 가 로드되지 않은 세션에서는 "수동 확인 필요" 로 넘기고 임의 단정 금지.
