## 3. 트랜잭션

- 여러 쓰기 작업이 묶이면 "전부 성공 또는 전부 롤백"을 보장한다. <예: Service 메서드에 @Transactional>
- 트랜잭션 경계는 **<어느 레이어>** 가 기본. <다른 레이어>에서 잡지 않는다.
- 신규 메서드는 동일 모듈의 기존 패턴을 먼저 참고.

## 4. 의존 방향

- <레이어 A> → <레이어 B> → <레이어 C> (단방향).
- 공통/외부 모듈(<예: 사내 프레임워크 jar>) 변경 금지. 필요 시 사용자 승인.

## 5. 패키지 / 디렉터리 구조

```
<소스 루트>/
├── <공통 모듈>/          # base, util, ...
├── <도메인 모듈 A>/
└── <도메인 모듈 B>/
```

각 도메인 모듈의 표준 하위 구조:

```
<module>/
├── <프리젠테이션 계층 파일>
├── <서비스 계층 파일>
└── <데이터 접근 계층 파일>
```

## 6. 신규 코드 작성 가이드

1. 유사 도메인의 기존 파일을 먼저 검색.
2. 동일한 네이밍/패키지 컨벤션 준수.
3. 기존에 없는 새 패턴 도입 시 본 문서에 추가 후 진행.
4. `.bak` / `.old` / `.nouse` 등 보관용 파일이 아니라 **동일 디렉터리의 활성 파일 패턴**을 따른다.

## 7. 금지 사항

- <레이어 건너뛰기 금지 규칙. 예: Controller 에서 직접 DB 접근 금지 — 반드시 Service 경유>
- <View → Service 직접 호출 금지 등>
- 유틸/헬퍼에 비즈니스 로직 두지 않는다.
