# DOC_GARDENING.md

> 문서를 코드와 함께 살아있게 유지하는 규칙. (하네스 유지보수의 메타 문서)

## 원칙

- **사실과 어긋난 문서 = 버그**. 발견 즉시 고친다.
- 짧고 정확하게. 의식의 흐름 금지.
- 같은 사실을 두 곳에 적지 않는다(중복 시 한쪽에서 링크).

## 문서 위치 규칙

| 위치 | 성격 | 예시 |
|------|------|------|
| `docs/` | **영구 가이드** — AI 가 매 세션마다 참조하는 운영 규칙·아키텍처·컨벤션. 시간이 지나도 유효. | `ARCHITECTURE.md`, `SECURITY.md` 등 |
| `docs/plans/` | **개인 plan 작업 노트** — 진행 중 작업 메모. **git 미추적**. 완료 시 내용 제거. → `PLAN_SYSTEM.md` | `#123-user-login.md` |

> 일시적 작업 노트는 git 추적 경로에 만들지 않는다. plan 단위 작업 메모는 `docs/plans/`(git 미추적)에서
> 관리하고 완료 시 비운다. 작업 노트 내용이 영구 규칙으로 굳어지면 `docs/` 의 적절한 카테고리 문서로 옮긴다.

## 문서 안정성 (변경 빈도)

| 문서 | 성격 | 변경 빈도 |
|------|------|-----------|
| `CLAUDE.md`, `AGENTS.md` | **안정 문서(앵커)** — 운영 규칙·진입점·Quick Reference | **드물게.** 규칙이 바뀌거나 특별한 경우에만 |
| `docs/` 하위 | 카테고리 가이드 — 디테일·패턴·컨벤션 | 패턴/규칙이 추가·변경될 때 수시로 |

- **`CLAUDE.md` / `AGENTS.md` 는 자주 건드리지 않는다.** 디테일이 늘어나면 `docs/` 하위로 보내고,
  앵커 문서에는 **링크 한 줄**만 남긴다.
- 두 문서를 수정할 때는 "규칙 변경 또는 특별한 사유"가 맞는지 먼저 자문한다. 단순 디테일이면 `docs/` 가 올바른 위치다.

## docs 편집 전 필수 점검

`docs/` 하위 문서를 새로 만들거나 수정할 때는 **편집 전·후에 반드시** 확인한다:

1. **기존 규칙과의 정합성** — 추가/변경 내용이 다른 문서(특히 `CLAUDE.md`, `AGENTS.md`, 관련 카테고리)의
   기존 규칙과 **다르거나 충돌하지 않는지** 대조한다. 다르면 추측으로 덮어쓰지 말고 사용자에게 확인.
2. **중복 여부** — 같은 사실이 이미 있는지 검색한다. 있으면 새로 적지 말고 한쪽에서 링크한다.

## 문서별 분량 가이드

| 문서 | 권장 분량 |
|------|----------|
| `CLAUDE.md` | 60~120줄 |
| `AGENTS.md` | **60줄 이하** (엄격) |
| `docs/` 카테고리 문서 | 100~200줄 |
| `docs/product-specs/<module>.md` | 모듈 크기에 비례, 길어지면 분할 |

## 갱신 트리거

- 새 패턴 도입 → ARCHITECTURE 등에 1줄 추가
- 잘못된 패턴으로 작성했다가 수정받음 → 즉시 해당 문서에 기록
- 의존성 업그레이드 → CLAUDE.md 표 갱신
- 새 모듈 → product-specs/index.md 갱신 + 스펙 파일 추가
- MCP 추가/제거 → TOOLS.md 갱신

## 가지치기 (gardening)

- 오래 손대지 않은 문서는 주기적으로 검토.
- 더 이상 유효하지 않은 규칙은 **삭제**(주석 처리 X).
- "TODO" 항목은 주기적으로 점검 — 안 할 거면 지운다.

## 검토 체크리스트

문서 수정 시:
- [ ] 코드와 일치하는가
- [ ] 기존 규칙(다른 문서 포함)과 다르거나 충돌하지 않는가 — 다르면 사용자 확인
- [ ] 같은 내용이 중복되지 않는가 (중복 시 링크로 대체)
- [ ] (`CLAUDE.md`/`AGENTS.md` 수정 시) 규칙 변경·특별한 사유가 맞는가
- [ ] 링크가 살아있는가
- [ ] 분량 가이드 준수하는가
