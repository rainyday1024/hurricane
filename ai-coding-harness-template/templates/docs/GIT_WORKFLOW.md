# GIT_WORKFLOW.md

> 브랜치, 커밋, push 절차. **AI는 사용자 명시 지시 없이 commit / push / 브랜치 삭제를 하지 않는다.**
> PR 생성·머지는 사용자가 직접 수행.

## 원칙

- 한 작업 = 한 브랜치 = 한 가지 목적 (버그 수정 + 리팩터링 섞지 않기).
- 커밋 메시지는 "왜"에 초점. "무엇"은 diff가 말해준다.
- 변경 단위는 사람이 한 번에 검토할 수 있는 크기로.

## 브랜치 네이밍

**규칙**: `<type>/<이슈번호>-<영문요약>`

- 이슈 트래커에서 이슈 번호를 확인 후 브랜치명에 붙인다.
  <!-- 이슈 트래커 MCP 가 있으면 여기서 언급하고 TOOLS.md 로 링크. -->
- 영문요약은 kebab-case (소문자 + 하이픈).
- 이슈가 없는 작업은 사용자에게 이슈 등록 여부를 먼저 확인.

**type**: `feature/` (신규) · `hotfix/` (버그) · `chore/` (설정/문서) · `refactor/` (리팩터링)

**예시**: `feature/#123-user-login` · `hotfix/#201-list-paging` · `chore/#210-upgrade-deps`

     예: develop_xxx 통합 브랜치 + xxx/<type>/<이슈>-<요약> 하위 브랜치, 매일 pull 규칙 등. -->

## 커밋 메시지

```
<이모지(선택)> [화면/기능요약] <type>: <한 줄 요약>
```

- `type`: feat / fix / chore / refactor / docs / test
- 본문(선택): 왜 이 변경을 했는지 1~3줄.

## 작업 흐름

> **시작 전 브랜치 확인 (필수)**: 코드 변경을 시작할 때 현재 브랜치가 보호·통합 브랜치
> (`main` / `develop` 등)이면, **작업 전 멈추고 이슈 번호 기준으로 새 작업 브랜치를 딸지 사용자에게 질문**한다.
> 이들 브랜치에서 직접 작업하지 않는다.

1. 이슈 번호 확인.
2. 브랜치 생성 (위 규칙대로).
3. 변경 → 작업 중 빠른 검증 반복, 커밋 전 완료 검증 통과.
4. 커밋 (메시지 규칙 준수).
5. push 는 **사용자 명시 지시 후**에 진행.
6. PR 생성·리뷰·머지는 **사용자가 직접** 수행.

## 배포 (생명주기)

- <배포 수단과 실제 상태를 솔직히. 예: CI 로 자동 / Jenkins 이나 현재 수동 / 없음>
- **AI 는 배포를 수행하지 않는다.** 배포는 사용자(운영) 영역.
- <PR 검증·자동 빌드 게이트(CI) 유무 → `FEEDBACK_LOOPS.md` 로 링크>

## 금지

- `--no-verify` 등으로 커밋 검증 우회 금지.
- `--force` push 금지 (특히 main).
- 사용자 승인 없는 `git reset --hard`, `git clean -f`, 브랜치 삭제 금지.
- 미사용 코드 / 주석 처리된 죽은 코드 포함 커밋 금지.
- 의도 없는 포맷 변경 (불필요한 diff) 금지.
- 사용자 지시 없이 자동 push / PR 생성 / 머지 금지.
