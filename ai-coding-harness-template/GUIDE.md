# AI 코딩 하네스 구축 가이드

> **목적**: AI 코딩 에이전트(Claude Code 등)가 우리 코드베이스에서 일관되고 안전하게 작업하도록 만드는
> "하네스(harness)"를 **어떻게 처음부터 구축·이식하는지**, 그 과정·결정·구조를 정리한다.
> **대상 독자**: 자기 프로젝트에 하네스를 도입하려는 개발자 / 팀에 새로 합류하는 사람.
>
> 이 문서는 *방법론·내러티브*다. 실제 운영 규칙의 정본은 각 문서(`CLAUDE.md`·`AGENTS.md`·`docs/`)에 있고,
> 여기선 링크로 연결한다.

---

## 1. 하네스란 무엇이고 왜 필요한가

AI 에이전트에게 "코드를 짜라"고만 하면, 프로젝트의 규칙·구조·맥락을 몰라 제멋대로 작성한다.
**하네스 = 에이전트를 우리 방식대로 일하게 하는 가이드레일의 묶음**이며, 일반적으로 4개 기둥으로 구성된다.

| 기둥 | 의미 | 대표 문서 |
|------|------|-----------|
| ① 문서화 (컨텍스트) | 프로젝트 규칙·구조·도메인 지식 | `CLAUDE.md`, `AGENTS.md`, `docs/*` |
| ② 아키텍처 제약 | 레이어링·모듈 경계·금지 패턴 | `ARCHITECTURE.md` |
| ③ 피드백 루프 | 실수를 즉시 잡는 자동 검증 (린터·테스트·훅) | `FEEDBACK_LOOPS.md` |
| ④ 생명주기 관리 | 빌드·배포·PR 자동화 | `GIT_WORKFLOW.md` |

> 핵심 통찰: 문서(①②)는 "읽고 해석해야" 적용되지만, 피드백 루프(③)는 어기는 **순간 빨간불**이 켜진다.
> 성숙도에 따라 ①②를 먼저 세우고, ③④는 프로젝트 상황에 맞춰 점진 도입해도 좋다 — 단 "있는 척"은 금지(§6).

## 2. 어떻게 시작하는가 (단계별)

### Phase 0 — 초기 골격

- **참고 자료로 초안 작성**: 하네스 관련 공개 문서·블로그를 참고해 골격을 잡는다(출처 → §8).
- 최소 세트 생성:
  - `CLAUDE.md` (운영 규칙 + 진입점), `AGENTS.md` (60줄 이하 업무 지침)
  - `docs/`: `ARCHITECTURE` / `PLAN_SYSTEM` / `SECURITY` / `GIT_WORKFLOW` 정도.
- **코드 대조 검토로 추정을 사실로 교정**: AI 초안에는 반드시 추측이 섞인다. 실제 코드·설정을 열어
  프레임워크 버전, 모듈 목록, 빌드 명령, 훅 존재 여부 등을 확인하고 어긋난 부분을 고친다.

> **권장 — 한 번에 다 만들지 말 것.** 초기엔 핵심 문서 **3~4개**(예: `CLAUDE.md`·`AGENTS.md`·`ARCHITECTURE.md`)만
> 잡고, 실제 작업하면서 필요한 문서를 **점진적으로 늘린다**. 처음부터 전부 채우려 하면 코드를 안 본
> 추측성 내용이 늘고 유지보수 부담만 커진다.

### Phase 1 — 정련 & 팀 배포 준비

초기 골격을 "팀에 배포 가능한 품질"로 끌어올리며 다음을 수행한다.

- **문서 거버넌스 강화** (`DOC_GARDENING.md`):
  - `CLAUDE.md`·`AGENTS.md` 는 **안정 문서(앵커)** — 규칙이 바뀌는 등 특별한 경우에만 수정, 디테일은 `docs/` 로.
  - `docs/` 편집 전 **기존 규칙 충돌·중복 점검** 의무화 (단일 진실원천).
- **작업 노트 체계 정리** — 진행 중 작업 메모는 별도 경로(예: `docs/plans/`, git 미추적)에서 관리하고
  완료 시 비우는 관례를 도입한다.
- **피드백 루프 명문화** (`FEEDBACK_LOOPS.md`) — 현황·결정사항·한계를 솔직히 기록한다(§6).
- **검증 명령 정립** — 예: `<빠른 검증 명령>`(작업 중 반복) / `<완료 검증 명령>`(완료 기준). 모든 문서에 일관 반영.
- **생명주기 현황 명시** — 배포·CI/PR 자동화의 실제 상태를 `GIT_WORKFLOW.md` 에 기록.
- **전체 문서 감사** — 사실 오류(존재하지 않는 훅을 전제한다든지)·중복·인덱스 불일치 정리, 분량 가이드 점검.
- **브랜치 가드 추가** — 작업 시작 시 현재 브랜치가 보호·통합 브랜치면 새 작업 브랜치를 딸지 질문하게 한다.

## 3. 표준 문서 구조 (지도)

```
CLAUDE.md            # 진입점 + 운영 규칙 (안정 문서)
AGENTS.md            # 60줄 이하 업무 지침 + Quick Reference (안정 문서)
docs/
├── ARCHITECTURE.md      # 레이어·모듈 경계·금지 패턴 (②)
├── PLAN_SYSTEM.md       # plan 작성 기준·형식·절차
├── FEEDBACK_LOOPS.md    # 피드백 루프 현황·결정 (③)
├── GIT_WORKFLOW.md      # 브랜치·커밋·배포(생명주기) (④)
├── UI_VALIDATION.md     # UI 변경 검증 절차 (수동/자동 피드백)
├── SECURITY.md          # 보안 규칙 (OWASP 기준)
├── TOOLS.md             # MCP / 외부 도구 사용 규칙
├── DOC_GARDENING.md     # 문서 유지보수 거버넌스 (메타)
└── product-specs/       # 모듈/도메인 스펙 카탈로그
```

프로젝트 스택에 따라 문서를 더한다. 예를 들어 프론트엔드 규칙이 두꺼우면 `FRONTEND.md`,
특정 그리드/컴포넌트 라이브러리를 쓰면 `<라이브러리>_GUIDE.md` 같은 **스택 특화 가이드**를 추가한다.
반대로 필요 없는 문서(예: UI 없는 백엔드 서비스의 `UI_VALIDATION.md`)는 뺀다.

## 4. 핵심 운영 원칙 (거버넌스)

팀 적용 시 가장 중요한, 문서를 "살아있게" 유지하는 규칙들:

- **단일 진실원천(SSOT)**: 같은 사실을 두 곳에 적지 않는다. 상세는 한 곳, 나머지는 링크.
- **안정 문서 최소 변경**: `CLAUDE.md`·`AGENTS.md` 는 자주 바꾸지 않는다. 디테일은 `docs/` 로.
- **분량 가이드**: `CLAUDE.md` 60~120줄 / `AGENTS.md` ≤60줄 / 카테고리 문서 100~200줄.
- **편집 전 점검**: 기존 규칙과 충돌·중복 여부를 먼저 대조, 다르면 사용자에게 질문.
- **사실 일치**: 코드/실태와 어긋난 문서는 버그로 간주, 발견 즉시 수정.
- 상세 규칙: [`docs/DOC_GARDENING.md`](templates/docs/DOC_GARDENING.md).

## 5. 표준 작업 워크플로

```
1. 작업 시작 — 읽기/준비
   CLAUDE.md → AGENTS.md → 관련 docs
   현재 브랜치가 보호·통합 브랜치면 → 이슈 번호 기준 새 브랜치 딸지 질문
2. 계획 (2파일 이상·새 모듈·마이그레이션 등 비단순 작업)
   PLAN_SYSTEM 따라 plan 작성 → ★사용자 승인★
3. 구현 — 단계별 코드 작성, 단계마다 빠른 검증 명령 실행
4. 완료 검증 — 완료 검증 명령 통과 (UI면 UI_VALIDATION 절차 병행)
5. 오류 시 — 통과할 때까지 수정 (4↔5 반복)
6. 마무리 — 작업 노트 정리, (사용자 지시 시) 커밋 / PR·머지·배포는 사용자
```

## 6. "도입하지 않은 것"도 명시하라 (성숙도 솔직함)

팀 공유 시 오해를 막으려면, *빠뜨린 것*과 *의식적으로 보류한 것*을 구분해 기록한다.
아래는 예시 표 — 자기 프로젝트의 실제 상태로 채운다.

| 항목 | 현 상태 | 사유 |
|------|---------|------|
| 정적분석 (린터) | 미도입 / 도입 | 예: 레거시 대량 위반 부담 |
| pre-commit hook | 미도입 / 도입 | 예: 현 단계 보류 (도입 시 비파괴 방식 권장) |
| CI/CD·PR 검증 자동화 | 없음 / 있음 | 예: 배포 수단은 있으나 현재 수동 운용 |
| 테스트 스위트 | N개 | 예: 0개면 `check`/`build` 는 사실상 컴파일 검증 수준 |

> 이 결정들을 `FEEDBACK_LOOPS.md` 에 기록하고, 도입 시 그 문서를 갱신한다. **"있는 척" 금지.**

## 7. 다른 프로젝트에 이식하는 법 (요약)

1. **핵심 3~4개부터 작게 시작** — `CLAUDE.md`·`AGENTS.md` 에 `ARCHITECTURE` 정도만 먼저. 한 번에 다 만들지
   말고 작업하며 늘린다(§2). 자매 프로젝트 구조 재사용 시 빠르다.
2. `docs/` 에 4기둥 문서를 배치하고 **코드와 대조** — AI 초안엔 추측이 섞이므로, 실제 코드·설정을 열어
   확인하고 사실과 다른 부분을 교정한다 (프레임워크 버전, 모듈 수, 훅 존재 여부 등).
3. `DOC_GARDENING.md` 의 거버넌스(SSOT·안정문서·분량·편집 전 점검)를 먼저 합의한다.
4. 피드백 루프(③)·생명주기(④)는 프로젝트 성숙도에 맞춰 **현황을 솔직히 기록**하고 점진 도입한다.
5. 개인 작업용 산출물(인수인계 노트, plan 노트)은 팀 배포 대상과 분리한다.
